# Поток данных и работа с данными (Data & Flow)

В данном проекте используется архитектурный подход **Clean Architecture** в сочетании с шаблоном **MVVM** и принципами **Unidirectional Data Flow (UDF)**. Ниже описаны принципы взаимодействия с сервером, устройство репозиториев и механизмы синхронизации состояния (на примере корзины и товаров).

---

## 1. Основные принципы

1.  **Разделение на слои**: Каждый слой имеет четкую зону ответственности и не зависит от деталей реализации других слоев.
2.  **Однонаправленный поток данных (UDF)**: Данные передаются только в одном направлении (от источника к UI), а события — в обратном (от UI к источнику).
3.  **Единый источник истины (SSOT)**: Каждое состояние имеет только один источник, ответственный за его изменение.

---

## 2. Архитектурные слои

### Слой данных (Data Layer)
*   **Ответственность**: Получение данных из сети (Retrofit), локальной БД (Room) или InMemory Store.
*   **Компоненты**: Repositories (реализация), DataSources (Remote/Local), DTO (Data Transfer Objects), Mappers.

### Доменный слой (Domain Layer)
*   **Ответственность**: Чистая бизнес-логика.
*   **Компоненты**: Use Cases, Repository Interfaces, Domain Models.

### Слой представления (Presentation Layer)
*   **Ответственность**: Отображение данных и обработка действий пользователя.
*   **Компоненты**: UI (Compose), ViewModels, UI State, UI Models.

---

## 3. Клиент-серверное взаимодействие и Репозитории

Взаимодействие с бэкендом строится на принципе **инкапсуляции сетевой логики** внутри слоя данных. Репозиторий реализует принцип **Single Source of Truth (SSOT)**.

### Принцип работы Репозитория:
UI подписывается на `Flow` из локального хранилища (Store), а не на разовые ответы сервера. При вызове действий (например, обновление корзины), репозиторий:
1.  Отправляет запрос через `RemoteDataSource`.
2.  Получает `DTO`, мапит его в `Domain Model`.
3.  Обновляет локальный `Store`.
4.  `Store` автоматически эмитит новое состояние всем подписчикам.

---

## 4. Управление состоянием корзины (Shared State)

Для таких сущностей, как **Корзина**, важно, чтобы данные были синхронизированы на всех экранах (список товаров, детали, экран корзины).

### Механизм синхронизации Shared State:

Общее состояние корзины реализуется через **реактивную подписку** всех заинтересованных экранов на единый источник данных:

1.  **Единый источник данных (SSOT)**: Внутри `CartRepository` находится `CartStore` (например, на базе `MutableStateFlow`). Он хранит актуальное состояние корзины в единственном экземпляре.
2.  **Реактивная подписка**: Экраны "Список товаров", "Детали товара" и "Корзина" используют `ObserveCartUseCase`, который предоставляет `Flow<Cart>` из репозитория.
3.  **Обновление данных**: Когда на любом из экранов пользователь меняет количество товара (через методы репозитория), `CartStore` обновляется.
4.  **Мгновенное уведомление**: Благодаря использованию `Flow`, обновление в `CartStore` автоматически рассылается всем активным подписчикам. Это гарантирует, что блок корзины внизу экрана и счетчики товаров на всех экранах в стеке навигации обновятся мгновенно и синхронно.

---

## 5. Расчеты и логика данных

Мы разделяем сырые данные и бизнес-информацию:

1.  **Data Layer**: Хранит только базовые поля (ID товара, цена, количество).
2.  **Domain Layer (Domain Model)**: Содержит логику расчетов. 
    *   `CartItem.lineTotal` = цена * количество.
    *   `Cart.summary` = общая сумма и общее количество товаров в корзине.
3.  **Presentation Layer (UI Model)**: Форматирует данные (например, "1 200 ₽").

### Сценарий использования (Flow):
*   **Экран списка**: Подписывается на `ObserveCartUseCase`. Для каждого товара отображается текущее количество в корзине. Блок внизу реактивно обновляется при любом изменении.
*   **Экран деталей**: При нажатии "Добавить" вызывает метод репозитория. Блок корзины внизу экрана обновляется мгновенно, так как использует тот же `ObserveCartUseCase`.
*   **Экран корзины**: Позволяет редактировать список. Любое изменение через репозиторий отразится на всех предыдущих экранах в стеке навигации.

---

## 7. Модульная структура фичи

Каждая функциональная фича (например, `cart`, `product_list`) разделена на несколько Gradle-модулей для обеспечения строгой изоляции слоев и ускорения сборки.

### Список модулей для фичи `{feature}`:

1.  **`feature:{feature}:{feature}_domain`**
    *   **Тип**: Pure Kotlin/Java (без зависимостей от Android/Hilt).
    *   **Состав**:
        *   `model/`: Доменные модели (бизнес-сущности).
        *   `repo/`: Интерфейсы репозиториев.
        *   `usecase/`: Классы бизнес-сценариев (Use Cases).
2.  **`feature:{feature}:{feature}_data`**
    *   **Тип**: Android Library.
    *   **Состав**:
        *   `repo/`: Реализация репозиториев из Domain слоя.
        *   `remote/`: Retrofit-интерфейсы (API), `RemoteDataSource`, DTO модели.
        *   `local/`: Room DAO, Entities, `LocalDataSource`.
        *   `mapper/`: Мапперы (`DTO -> Domain`, `Entity -> Domain`).
3.  **`feature:{feature}:{feature}_presentation`**
    *   **Тип**: Android Library (Compose).
    *   **Состав**:
        *   `ui/`: Compose-экраны, компоненты, превью.
        *   `vm/`: ViewModels.
        *   `model/`: UI-модели и UI-стейт (UiState).
        *   `navigation/`: Определения NavKey и провайдеры записей навигации.
4.  **`feature:{feature}:{feature}_di`**
    *   **Тип**: Android Library (Hilt).
    *   **Состав**:
        *   Hilt-модули (`@Module`, `@InstallIn`), которые связывают реализации из `data` с интерфейсами из `domain`.
    *   **Зачем**: Позволяет модулю `presentation` зависеть только от `domain` (через интерфейсы) и `di`, скрывая детали реализации из `data`.

### Схема зависимостей между модулями:
*   `presentation` -> `domain`, `di`
*   `data` -> `domain`
*   `di` -> `data`, `domain`
*   `app` -> все `presentation` и `di` модули
